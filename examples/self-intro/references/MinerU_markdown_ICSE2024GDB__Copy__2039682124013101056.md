# ABSTRACT

Graph Database Management Systems (GDBMS), which utilize graph models for data storage and execute queries via graph traversals, have seen ubiquitous usage in real-world scenarios such as recommendation systems, knowledge graphs, and social networks. Much like Relational Database Management Systems (RDBMS), GDBMS are not immune to bugs. These bugs typically manifest as logic errors that yield incorrect results (e.g., omitting a node that should be included), performance bugs that lead to sub-optimal performance (e.g., long execution time caused by redundant graph scanning), and exception issues (e.g., system crashes or missing exceptions). 

This paper introduces Equivalent Query Rewriting (EQR), a novel, widely-applicable methodology for finding bugs in GDBMS. The core idea is to rewrite a GDBMS query into equivalent ones that trigger distinct query plans, and check whether they exhibit discrepancies in system behaviors. To facilitate the realization of EQR, we propose a general concept called Abstract Syntax Graph (ASG), which embeds the semantics of a query in the paths of a graph. Given a base query, an ASG will be constructed and then an equivalent query can be generated by finding paths collectively carrying the complete semantics of the base query. To this end, we further design Random Walk Covering (RWC), a simple yet effective path covering algorithm. As a practical implementation of these ideas, we develop a tool GRev, which has successfully detected 21 previously unknown bugs across 5 popular GDBMS, with 10 of them being confirmed. In particular, 14 of the detected bugs are related to improper implementation of graph data retrieval in GDBMS, which is challenging to be identified by existing techniques.

However, detecting bugs in GDBMS is very challenging because of the following reasons. First, unlike RDBMS which uniformly utilize SQL to access data, different GDBMS adopt different query languages, such as Neo4j’s Cypher [27], TinkerGraph’s Gremlin [35], and NebulaGraph’s nGQL [26]. Thus, existing differential testing tools, like GDsmith [18] and Grand [45], can only test the shared functionalities of GDBMS that adopt the same query language. Moreover, the adoption of different query languages can lead to extensive engineering effort when developers try to generalize existing tools to more query languages. We think this is the main reason why existing research only focuses on at most two query languages: Cypher and Gremlin. 

Second, it is non-trivial to adapt RDBMS testing approaches [2, 32, 33, 37] to GDBMS because it incurs re-implementation under different database models and query languages. Moreover, this kind of adaptation can only test the shared functionalities between RDBMS and GDBMS, leaving graph-related bugs underexplored. For example, GDBMeter [14], the only existing metamorphic testing approach for GDBMS, utilizes an RDBMS testing approach Ternery Logic Partitioning (TLP) [32], which partitions a query $Q$ into sub-queries ????, ??¬??, $Q _ { \mathrm { n u l l } }$ based on three possible outcomes of the predicate $P$ : True, False, and Null. Thus, although GDBMeter successfully found common bugs related to predicates handling, none of their reported bugs are graph-related. 

Third, performance bugs in GDBMS have not been carefully considered by existing approaches. Specifically, differential testing approaches, such as GDsmith [18] and Grand [45], fall short of detecting performance bugs because the diverse underlying architectures of different GDBMS can easily cause performance differences. The metamorphic testing approach, GDBMeter [14], is based on the relationship between the original query and three sub-queries, which do not exhibit clear performance relationships. Thus, it is difficult for existing approaches to find performance bugs. 

To tackle these challenges, this paper introduces Equivalent Query Rewriting (EQR), a novel, widely-applicable methodology for testing GDBMS. The core idea is inspired by Query Rewriting [28, 30], a concept in the database field aimed at (often rulebased) query optimization. Our idea is to rewrite a GDBMS query into equivalent ones that trigger distinct query plans yet should return identical results. For example, the first query in Listing 1 was re-written to generate the second query as the equivalent query. If the returned results are different or a large difference was observed in the query time, a potential bug is found. To facilitate the realization of EQR, we propose a general, essential concept called Abstract Syntax Graph (ASG). The key insight is to embed the complete semantics of a query into the paths of a graph (i.e., ASG), which can be further utilized to generate equivalent queries by covering different sets of paths in the graph. Intuitively, ASG provides an abstraction for a query that can be mapped to various concrete graph models adopted by different GDBMS and their query languages. Thus, to generate an equivalent query that carries the complete semantics encoded in ASG, we need to find a set of non-overlapping paths that collectively cover all the relationships (i.e., edges) and constraints (i.e., labels and properties). This path-finding process is further handled by our proposed Random Walk Covering (RWC), a simple yet effective algorithm. Each set of the non-overlapping paths returned by RWC will be translated into an equivalent query. RWC can generate a large number of equivalent queries that trigger diverse query plans, and thus effectively stress-test GDBMS. 

We have implemented these ideas as a tool called GRev, and employed it to test five popular GDBMS, namely Neo4j [25], RedisGraph [31], MemGraph [23], TinkerGraph [42], and Nebula-Graph [24], which collectively adopt three different query languages: Cypher [27], Gremlin [35], and nGQL [26]. As of the time of writing, GRev has discovered 21 previously unknown bugs in these GDBMS, including 14 graph-related bugs, of which 10 have been confirmed. Among the 21 bugs, 12 are logic bugs, 3 are performance bugs, 4 are missing exceptions, and 2 are unexpected exceptions.

• To realize EQR, we propose Abstract Syntax Graph (ASG), a general concept that encodes the complete semantics of a query in a set of paths in a graph. 

• We design a simple yet effective algorithm Random Walk Covering (RWC) to generate equivalent queries from ASG. 

• We implement these ideas as a tool called GRev and used it to test 5 widely-used GDBMS, and successfully detected 21 previously unknown bugs. 14 of these bugs are graph-related bugs, with 9 of them being confirmed.

# 2 BACKGROUND

# 2.1 Labeled Property Graph in GDBMS

Most GDBMS adopt a labeled property graph (LPG) to store and query data. As shown in Fig. 1, an LPG contains nodes (e.g., n1, n2 and n3), relationships (e.g., r1 and r2), corresponding labels (e.g., n1 : person and r1 : read), and properties (e.g., name : Alice) for each node and relationship. The flexible graph structure can efficiently capture interconnected information regarding entities, relationships, and attributes, whereas RDBMS stores this information using several tables. Owing to its flexibility and effectiveness in representing interconnected data, the LPG model has become a crucial component of GDBMS. 

For example, there are three nodes (n1, n2, and n3) and two relationships (r1 and r2) in Fig. 1. Among these nodes, n1 with label person has properties name and age; n2 with label book has properties name and language; and n3 with label person has properties name and birth. To describe the connections between these nodes, two directed relationships are involved, which are r1 with label read and r2 with label write. 

# 2.2 Query Languages in GDBMS

Unlike RDBMS, which utilizes SQL for creating, modifying, and retrieving data, graph databases do not have a standardized query language. Instead, multiple query languages have been developed for different GDBMS to cater to various application requirements. For instance, Cypher [27] is a popular SQL-like query language that employs SQL-like clauses such as MATCH, WHERE, and RETURN to retrieve data. On the other hand, Gremlin [35] is a widely-used functional query language that expresses queries in a data-flow manner. Among the top 10 GDBMS according to the DB-Engine Ranking [6], 6 of them support Gremlin or Cypher as their query language. Therefore, we realize EQR for mainly validating Cypher-based and Gremlin-based GDBMS. In addition, we tested NebulaGraph [24] that utilizes their own language nGQL [26] to further verify the effectiveness and generalizability of our approach. 

Cypher. Cypher [27], designed by Neo4j [25], is a concise and powerful query language that leverages patterns to express the data for retrieval, and SQL-like clauses to manage and manipulate the data. A pattern defines a set of abstract nodes and relationships to match, represented by (var) and -[var]-, respectively. Specifically, var represents optional variables assigned to each abstract node and relationship, which are associated with constraints on label, property, and path. For example, in the Cypher query MATCH 

(n3:person)-[r2:write]- $\cdot >$ (n2:book) WHERE (n2.name = "Ani mal Farm") RETURN n3.name, MATCH is an SQL-like clause used to express matching in Cypher. It is followed by a pattern that consists of two nodes and one directed relationship (represented by an arrow $\ ' > \implies$ ), along with their corresponding variables (e.g., n2, and r2) and constraints on labels (e.g., r2 : write). WHERE and RETURN are also two SQL-like clauses in Cypher. WHERE clause is used to filter the matched patterns that have the property name: Animal Farm, while RETURN clause is used to return the resulting data. This Cypher query can be used to find the author of the “Animal Farm” in the graph shown in Fig. 1. 

Gremlin. Gremlin[35], developed by TinkerPop[43], empowers users to write complex traversals in a functional and data-flow manner. A Gremlin query consists of a sequence of Gremlin steps, each serving as a building block of the traversals and designed to perform specific operations on the data. These steps can be utilized to map elements in the graph, filter data, perform aggregations, or modify the graph structure. For example, in the Gremlin query g.V().has("name", eq("Alice")).out("read"), the first step, V(), selects all nodes in the graph. The second step filters the nodes that have the property name: Alice, and the third step traverses to its neighboring node through an outward relationship called read. This Gremlin query can be used to find the book that Alice read in the graph shown in Fig. 1. 

# 2.3 Query Rewriting

Query rewriting [30] is a crucial step in RDBMS query optimization, which primarily aims at transforming an original query into an equivalent but more efficient form to speed up execution. This transformation is facilitated by a set of hardcoded rewrite rules, which are selected and applied using a production rule engine, significantly improving the efficiency of query execution. 

While inspired by Query Rewriting, our EQR pursues a different goal, which is to generate numerous equivalent queries with diverse query plans without any hardcoded rules. In addition, existing query rewriting rules explored in the database field cannot be used in GDBMS testing because of the following reasons. First, most query rewriting studies focus on RDBMS and there is a lack of similar techniques in GDBMS. Second, even if researchers start to explore rewriting rules in graph databases for query optimization, which is a challenging research direction, the equivalent queries generated by them are unlikely to trigger diverse query plans, because these potential rules would be quickly integrated into the optimizers of graph databases and the equivalent queries generated by them will be compiled into the same query plan. Differently, our proposed EQR is a general methodology for validating GDBMS (different goal) and the equivalent queries generated by our approach aim to trigger diverse query plans.

# 3.1 Abstract Syntax Graph (ASG)

Various graph database query languages can offer distinct functionalities, cater to different usage scenarios, and employ unique syntax. This diversity presents a significant challenge when attempting to propose a method that supports rewriting equivalent queries for different query languages universally. Despite this challenge, graph databases generally share the same fundamental goal of searching for graph components that have certain relationships between nodes. For example, identifying nodes that form a triangle, which can be written as the query MATCH (n1)-[]-(n2)-[]-(n3)-[]-(n1 ) in Cypher, or g.V().as("n1").both().as("n2").both().as(" n3").where(both().as("n1")) in Gremlin. 

Based on this insight, we propose ASG, a general concept that can represent the semantics of a query, i.e., relationships between nodes to be retrieved by the queries and constraints around the edges or nodes, in a language-independent manner. ASG is not specifically tailored to any particular query language, which makes it a flexible and generic representation for various queries. Specifically, to extend our tool to a new graph database, developers only need to provide a parser that translates between the query language and ASG, rather than writing full-stack code from scratch. 

Given a query ??, an ASG is a graph structure that encapsulates the relationships between nodes to be retrieved by ??. In particular, these relationships are defined as patterns in Cypher [27] and traversals in Gremlin [35]. In a rigorous formalization, the ASG of $Q$ is denoted as $G ( Q ) = \left( V , E \right)$ , where 

• $V$ represents a set of vertices. Each vertex corresponds to a category of nodes to be retrieved by $Q$ , aligned with their specific constraints defined in ?? (e.g., they must carry certain labels, or their property values have to satisfy a certain inequality). 

• ?? is a set of edges, where each edge links two vertices within ?? , describing the relationship that should be satisfied between these two categories of nodes. 

For instance, consider the ASG in Fig. 2, which illustrates the relationships between the nodes to be retrieved. This ASG comprises four vertices, namely v1, v2, v3, and v4. These vertices correspond to four categories of nodes—n1, n2, n3, and n4—to be retrieved, each paired with specific constraints. As an example, v1 aligns with the category of nodes labeled as person, with the name property being “Alice”. Additionally, there are three edges in the ASG: $\mathsf { v } \mathsf { 1 } \longrightarrow \mathsf { v } \mathsf { 2 }$ , $\mathsf { v } 2 \gets \mathsf { v } 3$ , and ${ \mathsf { v } } 2 { \longrightarrow } { \mathsf { v } } 4$ . These edges illustrate the relationships between the categories of nodes. For example, the edge $\mathsf { v } \mathsf { 1 } \longrightarrow \mathsf { v } \mathsf { 2 }$ indicates that n1 should know n2 for each tuple (n1, n2, n3, n4) among the retrieved data. 

# 3.2 ASG Parsing

The goal of ASG parsing is to convert the data relationships of queries, as expressed in the query languages, into a graph representation within the ASG. To facilitate this process, we only need to design lightweight (around 260 lines of Python code for Cypher [27]) ASG parsers customized for each query language. These parsers translate the diverse syntax and semantics of different query languages into a unified ASG representation. This translation also allows for the potential division of a complex query into multiple ASGs, where each ASG represents a distinct section of the query. 

In our implementation, for Cypher queries, each pattern within the query is extracted and converted into an individual ASG; and for Gremlin queries, the start step, end step, and each as() step are converted into ASG vertices, while the inner traversal steps are transformed into ASG edges. 

# 3.3 Random Walk Covering

Given the necessity of expressing queries in linear text, relationships that need to be retrieved are commonly represented by multiple path-like relationships within a GDBMS query. For instance, a Cypher query’s graph pattern consists of a series of path patterns, while a Gremlin query’s traversal is made up of a series of linear sub-traversals. 

Recognizing this, we can transform an ASG into a sequence of non-overlapping paths, which covers the ASG (i.e., every relationship and constraint within the ASG is represented by these paths exactly once), thereby corresponding to an equivalent GDBMS query. For example, as illustrated in Fig. 3, the ASG can be represented by various path sequences. In these sequences, each path (represented by a different color) matches a subset of the relationships and constraints described by the ASG. Collectively, these paths represent all the information present in the ASG by covering all its vertices and edges. These path sequences are then translated into GDBMS queries, resulting in equivalent queries generated from the ASG.

# 4.2 Graph Generation

Unlike some testing techniques [14, 18] that are specific to particular data, our EQR approach can be generally applied to any graph. 

We can simply create a random LPG graph schema on the target graph database by executing the following steps. 

• First, create properties and assign them unique names and data types (e.g., int, float, and string). 

• Second, create labels for nodes and relationships, with each label corresponding to a set of randomly selected properties. 

• Third, create nodes and relationships, assigning them random labels and properties, and then assigning random values to their properties based on their data types.

# 4.4 Equivalent Query Rewriter and Validation Rules

After generating the graph schema and the base query, GRev employs the aforementioned EQR approach, as described in Section 3, to generate multiple equivalent queries. For a pair of equivalent queries $( Q _ { a } , Q _ { b } )$ , we denote their results as $R ( Q _ { a } )$ and $R ( Q _ { b } )$ , and their execution times as $T ( Q _ { a } )$ and $T ( Q _ { b } )$ respectively. We will detect the potential bugs using the following validation rules, where $E _ { x }$ is defined as the set of exceptions that the GDBMS may return. 

Detecting Logic Bugs. Logic bugs represent a notorious category of bugs that can lead to incorrect query results without triggering any exceptions or warnings. In GRev, logic bugs can be detected by comparing the results between the two equivalent queries. For a pair of equivalent queries $( Q _ { a } , Q _ { b } )$ , if Equation 1 holds, it will indicate logic bugs in the GDBMS, as two equivalent queries produce different results without throwing any exceptions. 

$$
R \left(Q _ {a}\right) \notin E _ {x} \wedge R \left(Q _ {b}\right) \notin E _ {x} \wedge R \left(Q _ {a}\right) \neq R \left(Q _ {b}\right) \tag {1}
$$

In addition, if $R ( Q _ { a } ) \neq R ( Q _ { b } )$ but exceptions are thrown, it does not suggest logic bugs. Instead, it points toward exception-related bugs, which include unexpected exceptions and missing exceptions. 

Detecting Performance Bugs. Performance bugs are also considered significant bugs in GDBMSs as they have a negative impact on query response times. In GRev, performance bugs can be detected by comparing the execution time between the two equivalent queries. For a pair of equivalent queries $( Q _ { a } , Q _ { b } )$ , if Equation $2 ^ { 3 }$ holds, it will indicate a performance bug in the GDBMS, as two equivalent queries result in a significant difference in their execution times. 

$$
\begin{array}{l} \frac {\operatorname* {m a x} \left(T \left(Q _ {a}\right) , T \left(Q _ {b}\right)\right) - \operatorname* {m i n} \left(T \left(Q _ {a}\right) , T \left(Q _ {b}\right)\right)}{\operatorname* {m a x} \left(T \left(Q _ {a}\right) , T \left(Q _ {b}\right)\right)} \geq C _ {1}, \\ \max  \left(T \left(Q _ {a}\right), T \left(Q _ {b}\right)\right) - \min  \left(T \left(Q _ {a}\right), T \left(Q _ {b}\right)\right) \geq C _ {2}, \tag {2} \\ \end{array}
$$

where $C _ { 1 } , C _ { 2 }$ are two constant thresholds.

# 5.4 Comparison with Existing Techniques

As shown in Table 3, there are three closest-related works in testing GDBMS, i.e., GDsmith [18], Grand [45], and GDBMeter [14], that can detect bugs other than crashes. Specifically, GDsmith and Grand are based on Differential Testing [39] while GDBMeter is based on Query Partitioning [32] (i.e., partitioning query into multiple equivalent sub-queries). Notably, we can not compare GRev with RDBMS testing tools, such as SQLancer [33]. Although GRev and these tools both test the bugs in database systems, their target databases have fundamental differences, in terms of data structures (e.g., graph model vs. relational model), query mode (e.g., traversals vs. joins), and languages (e.g., Cypher [27], Gremlin [35], and nGQL [26] vs. SQL). 

Comparison with GDsmith and Grand. As differential testing tools, GDsmith [18] and Grand [45], the core idea behind them is to execute the same queries on multiple GDBMS and reveal bugs by discrepancies. However, these tools are limited to specific query languages and face challenges when it comes to broader testing scenarios. For example, GDsmith can only test Cypherbased GDBMS, while Grand is limited in testing Gremlin-based GDBMS. 

Even for testing the GDBMS that adopt the same query languages, different GDBMS can have their unique design or adopt some shared libraries with other GDBMS. In the first case, these differential testing tools are sensitive to queries or are prone to give false positives. For example, MemGraph [23] and Neo4j [25] will deliberately return different results for the same query MATCH ()-[R]-(), and thus GDsmith does not test such queries to avoid false alarms. In the second case, these tools are prone to give false negatives. For example, almost all Gremlin-based GDBMS share a codebase inherited from Apache TinkerPop [43] and TinkerGraph [42] (i.e., Gremlin library), resulting in the bugs (e.g., the logic inconsistency shown in Listing 5) in these shared codes being difficult to detect by Gremlin. 

In addition, GDsmith and Grand face challenges in detecting performance bugs by comparing performance discrepancies, as 

different target GDBMSs have varying underlying architectures. For instance, Neo4j is a distributed GDBMS, while MemGraph is an in-memory GDBMS. As a result, the same query may exhibit significantly different performances in these GDBMS. 

Based on these limitations and our manual analysis, GRev has identified at least 9 bugs that theoretically could not be detected by GDsmith, and at least 15 bugs that theoretically could not be detected by Grand. 

Comparison with GDBMeter. Based on Query Partitioning, GDB-Meter [14] leverages the test oracle called Ternary Logic Partitioning (TLP) to detect bugs in GDBMS. TLP [32] was initially introduced in testing RDBMS. It involves partitioning a query ?? into sub-queries $Q _ { P } , Q _ { \lnot P } , Q _ { \mathrm { n u l l } }$ based on three possible outcomes of the predicate $P$ : True, False, and Null. 

However, this partitioning approach is sensitive to query and query language. For example, where the query includes functionalities such as removing duplicate elements, aggregating the subqueries may no longer produce an equivalent result to the original query. As a result, GDBMeter is unable to effectively test queries that involve DISTINCT clauses in Cypher [27] and dedup() clauses in Gremlin [35]. This limitation makes it difficult to identify bugs like the one shown in Listing 5 using GDBMeter. Similarly to Grand [45] and GDsmith [18], GDBMeter also lacks general support for multiple query languages in GDBMS. For example, since Gremlin lacks steps to check for null values in expressions, GDBMeter can only test Gremlin-based GDBMS when working with graphs that do not contain Null values. 

In addition, TLP is not effective in detecting GDBMS’s performance bugs and graph-related bugs. Firstly, there are no inherent performance relationships between the three sub-queries and the original query, making it difficult for GDBMeter to detect performance bugs based solely on their execution times. Liu’s work [21] includes a comprehensive evaluation of TLP’s effectiveness in detecting RDBMS’s performance bugs, revealing that only one performance bug in RDBMS was triggered by TLP out of 2,000 test cases. On the other hand, GDBMeter, when applied TLP to test GDBMS, has not been successful in identifying any performance bugs. 

Secondly, TLP can only be applied to filter clauses (e.g., WHERE), and is not capable of effectively modifying query plans of the traversal process (e.g., MATCH), resulting in GDBMeter facing challenges in finding traversal bugs. # 6 DISCUSSION

Threats to Validity. First, while GRev tests GDBMS automatically, bugs require manual reduction and reproduction, possibly introducing errors. To mitigate this, three individuals carefully study both the original bugs and the reduced bugs and reach a consensus before reporting them to the developers. Second, given the diversity of GDBMS query languages, testing GRev’s universality across all isn’t feasible. To address this limitation, we focus on the three most popular query languages based on the DB-Engine Ranking. 

Future Directions. In this paper, we introduce ASG, a novel framework that represents the semantics of GDBMS queries as simple graph structures. By doing so, we can harness the inherent properties of graphs to construct test oracles, thereby simplifying the process. This leads to a focus on the graph alone, minimizing the need to navigate the multifaceted syntax of different GDBMS querying languages. Our EQR approach demonstrates the effectiveness of using ASG as a foundation for an efficient test oracle in this study. Looking forward, we believe that this versatile framework has the potential to significantly contribute to the development of comprehensive test oracles for GDBMS testing. 

# 7 RELATED WORK

Differential Testing. Differential testing is a widely employed technique for bug detection in systems. The fundamental idea behind it [10, 22] is to verify that, for a given input, different systems produce identical execution results. Leveraging this principle, differential testing effectively identifies inconsistencies between systems and finds practical applications in diverse research domains, including binary program analysis [29], desktop hardware evaluation [11], JVM investigations [4, 5], debugger assessments [17], AI systems examination [1, 7–9, 20], object-relational mapping systems [38], etc. Grand [45] and GDsmith [18] represent two established and prominent differential testing tools specifically designed for GDBMS. However, they can not support different query languages, provide test oracles for the single target GDBMS, and detect shared bugs across multiple GDBMS. In this work, our EQR approach addresses these challenges by providing a universal framework ASG and a test oracle based on metamorphic testing. 

Metamorphic Testing. Metamorphic testing [36] acts as a test oracle similar to differential testing. When testing the same system, new inputs can be generated by applying metamorphic transformation rules, and both the original and new inputs should produce the same output in the same system. This technique has been applied successfully in various domains [12, 19, 44]. For example, Orion [15], Athena [16], and Hermes [40] introduce metamorphic testing techniques in compilers such as equivalent dead code mutation and equivalent live code mutation to identify logic bugs within the compiler. Based on metamorphic testing, GDBMeter [14] is the SOTA tool for detecting logic bugs in GDBMS. However, none of the graph-related bugs has been detected by it due to reusing TLP[32], a metamorphic relation for RDBMS. In this work, we propose a novel and more effective metamorphic testing method for GDBMS that can both detect graph-related bugs and common bugs shared with RDBMS. 

# 8 CONCLUSION

This paper introduces three key ideas: (1) Equivalent Query Rewriting (EQR), a novel methodology for validating GDBMS; (2) Abstract Syntax Graph (ASG), a general concept facilitating EQR across different GDBMS and query languages; and (3) Random Walk Covering (RWC), a simple but effective algorithm for finding paths in an ASG. As a practical realization of these ideas, we developed a tool called GRev. GRev has been proven effective on 5 popular GDBMS, uncovering 21 previously unknown bugs, including logic bugs, performance bugs, and exception issues. In particular, GRev successfully found 14 bugs that are related to improper implementation of graph data retrieval in GDBMS, which is different from existing techniques that mainly report bugs related to similar functionalities in RDBMS. In addition, the GDBMS tested collectively adopt three different query languages, which is the most compared to existing research that considers one or two query languages.
